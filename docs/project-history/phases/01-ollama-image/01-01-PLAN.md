<!-- Key decisions governing this file. All are locked (non-negotiable). -->

D-01: File location — Dockerfile.ollama at repository root alongside Dockerfile
D-02: One RUN layer per model. Pattern:
  OLLAMA_HOST=127.0.0.1:11235 ollama serve &
  _pid=$! && _ready=0
  for i in $(seq 1 30); do
    if OLLAMA_HOST=127.0.0.1:11235 ollama ls >/dev/null 2>&1; then _ready=1; break; fi
    sleep 2
  done
  [ "$_ready" = "1" ] || (echo "ERROR: ..." && kill $_pid && exit 1)
  OLLAMA_HOST=127.0.0.1:11235 ollama pull <model>
  kill $_pid && wait $_pid 2>/dev/null || true
D-05: ENV OLLAMA_HOST=0.0.0.0:11434 — set AFTER all model pull RUN layers
D-07: GPU/CPU detection is runtime — no GPU-specific directives in Dockerfile
D-08: No device_requests — GPU config is Phase 2 (compose), not Phase 1 (Dockerfile)
D-09: GHCR 10GB-per-layer limit — document in Dockerfile header comment

From research:
- Base image: ollama/ollama:0.20.3 (pin exact semver — latest tag lags)
- curl is NOT in base image — must install: apt-get install -y -qq curl
- Build-time port: 127.0.0.1:11235 (avoid collision with final ENV 0.0.0.0:11434)
- curl is absent from base: github.com/ollama/ollama/issues/9781
- ollama ls is the correct readiness probe (no curl inside container during build)
- mode=max required on cache-to (model layers are intermediate, not final stage)
</interfaces>

   - Published to: ghcr.io/ldj-share/.dotfiles/ollama-models:latest

   FROM ollama/ollama:0.20.3

   RUN apt-get update -qq && apt-get install -y -qq curl && rm -rf /var/lib/apt/lists/*

4. Pull gemma4:26b — one RUN layer (per D-02, independent GHA cache scope ollama-gemma4-26b):
   Comment: "Pull gemma4:26b (~18GB) — GHA cache scope: ollama-gemma4-26b"
   Use build-time port 127.0.0.1:11235 (NOT 11434 — avoids collision with final ENV)
   Full retry loop (30 retries, 2s sleep) using `ollama ls` as readiness probe
   Exit 1 with descriptive error if server does not start within 60s
   Pull: OLLAMA_HOST=127.0.0.1:11235 ollama pull gemma4:26b
   Clean shutdown: kill $_pid && wait $_pid 2>/dev/null || true

5. Pull gemma4:e4b — separate RUN layer (per D-02, independent GHA cache scope ollama-gemma4-e4b):
   Comment: "Pull gemma4:e4b (~9.6GB) — separate RUN = independent cache layer (D-02)"
   Same retry loop pattern as above
   Pull: OLLAMA_HOST=127.0.0.1:11235 ollama pull gemma4:e4b
   Same clean shutdown

6. Set runtime binding AFTER all model pulls (per D-05, after all RUN layers):
   ENV OLLAMA_HOST=0.0.0.0:11434

7. Expose port:
   EXPOSE 11434

8. HEALTHCHECK using curl (installed in step 3):
   HEALTHCHECK --interval=30s --timeout=10s --start-period=60s --retries=3 \
     CMD curl -sf http://localhost:11434/api/tags || exit 1

No CMD override needed — base image CMD is `ollama serve`, which is correct.
No GPU directives anywhere — GPU is runtime/Phase 2 concern (D-07, D-08).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Dockerfile → base image registry | Pulls FROM ollama/ollama — if tag is mutable (:latest), supply chain substitution is possible |
| CI runner → GHA cache | BuildKit restores cached layers from GHA; cache is repository-scoped but could contain stale/poisoned data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-01 | Tampering | `FROM ollama/ollama` base image | mitigate | Pin `FROM ollama/ollama:0.20.3` — exact semver prevents mutable tag substitution; tag is verified in acceptance criteria |
| T-1-03 | Tampering | GHA cache restore (model RUN layers) | accept | GHA cache is repository-scoped; only collaborators with write access can poison it; acceptable for this threat model |
</threat_model>
